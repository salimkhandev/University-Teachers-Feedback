ABSTRACT

The University Teachers Feedback System is a comprehensive, AI-powered web application designed to streamline the collection, analysis, and visualization of student feedback regarding university faculty members. The system provides a digital platform where students can anonymously submit feedback on their assigned teachers through an intuitive user interface, administrators can monitor the entire ecosystem, and teachers can receive AI-generated summaries, interactive rating visualizations, and personalized chatbot coaching based on actual student comments. The system employs a modern technology stack comprising React 18 with Vite for the frontend, Node.js with Express for the backend, MongoDB with Mongoose ODM for data persistence, and Google Gemini AI (gemini-2.5-flash / gemini-2.5-flash-lite models) for natural language processing tasks. Role-based access control (RBAC) ensures that students, teachers, and administrators each have appropriate views and permissions. The platform incorporates intelligent caching strategies to optimize AI API usage, multi-key rotation with automatic invalid key detection and deactivation, CSV bulk import capabilities for rapid data population, JWT-based authentication with access/refresh token rotation, user account CRUD with cascading deletion of related data, AI settings management panel for model selection and key lifecycle operations, admin-teacher chat with smart history summarization, and comprehensive analytics dashboards with statistical charts. This thesis presents the complete system architecture, database schema design across 11 Mongoose models, implementation decisions, feature specifications, testing strategies, and deployment considerations of this production-ready educational technology solution.

CHAPTER 1: INTRODUCTION

1.1 Overview

In the contemporary landscape of higher education, the quality of teaching directly influences student learning outcomes, institutional reputation, and overall academic excellence. Universities worldwide have recognized the critical importance of collecting and analyzing student feedback as a mechanism for continuous improvement in teaching quality. However, the process of gathering, processing, and deriving actionable insights from student feedback has traditionally been fraught with challenges spanning administrative, pedagogical, and technological domains. The University Teachers Feedback System is an AI-powered web-based solution designed to automate and enrich this process. Utilizing React 18 with Vite for the frontend, Node.js with Express and TypeScript for the backend, MongoDB with Mongoose for data persistence, and Google Gemini large language models for natural language processing, the system provides a comprehensive digital platform where students can anonymously evaluate their assigned teachers, faculty members can receive detailed analytics with AI-generated text summaries, and administrators can manage the entire ecosystem through a powerful configuration dashboard.

1.2 Motivation

Many universities have traditionally relied on paper-based feedback forms distributed at the end of each academic term. These systems present numerous inefficiencies: they are resource-intensive to print and distribute, require manual data entry, are susceptible to data loss, and provide limited anonymity which discourages honest student responses. The logistical burden of managing thousands of paper forms across multiple departments is substantial. Even when institutions transition to basic digital solutions, many systems offer little more than simple form submission and numerical rating aggregation. These systems fail to extract meaningful insights from qualitative comments, which constitute the richest source of feedback data. The inability to automatically analyze text-based feedback represents a significant missed opportunity, as student comments contain nuanced observations about teaching methodology, communication effectiveness, and classroom management that cannot be captured through numerical ratings alone. The motivation behind this project is to build an intelligent, secure, end-to-end automated platform that leverages Google Gemini models to solve both quantitative and qualitative feedback bottlenecks. The system transforms raw student comments into coherent, unbiased, actionable insights for faculty, while providing administrators with dashboards for institutional monitoring and AI conversation tools for in-depth teacher evaluation.

1.3 Problem Statement

Educational institutions face several critical challenges in implementing effective teacher feedback systems: 1. Lack of integrated platforms: Institutions often use disjointed tools — students submit on one form, admins aggregate in Excel, teachers receive static PDF reports weeks later — leading to data inconsistencies and delays. 2. Manual qualitative analysis: Processing hundreds of student comments to extract meaningful patterns requires prohibitive human effort, so qualitative data is often left unanalyzed. 3. Broken feedback loop: Students do not see visible impact of their feedback, and teachers receive raw feedback dumps without structured constructive guidance. 4. Complex system configuration: Managing university organizational structures (departments, semesters, sections, subjects, teacher assignments) manually is error-prone. 5. AI API reliability: Integrating with external AI services introduces challenges around rate limits, key rotation, caching, and error handling that must be systematically addressed.

1.4 Proposed Solution

The proposed solution is a modular, responsive Single Page Application integrated with a secure RESTful API (Express with TypeScript) and a Generative AI backend (Google Gemini). The solution is tailored for three distinct user roles with the following actual implemented features: Students: Access a portal displaying their assigned teachers and subjects based on section/semester enrollment. Submit ratings (1-10) and written comments through an intuitive feedback form. Edit their submissions with version tracking (incrementing version number with history preserved in database). View progress tracking showing how many teachers they have rated out of total assigned. Teachers: Access a comprehensive dashboard featuring Recharts-based visualizations of rating distributions and subject averages. View AI-generated text summaries of student comments with a "stale" indicator when new feedback arrives. Interact with an AI Chat Coach — a Gemini-powered chatbot primed with their actual feedback data that offers personalized, constructive teaching guidance. The chat system maintains conversation context with smart history management. Administrators: Oversee the platform through an overview dashboard with system statistics (total teachers, students, feedback count). View teacher rankings sorted by average rating with bar charts. Track student feedback completion status with missing teacher names per department using pagination. View pending students list across all departments showing which teachers they haven't rated. Manage user accounts with full CRUD operations including password reset and cascading deletion. Configure AI settings: switch between Gemini models (gemini-2.5-flash, gemini-2.5-flash-lite), add/activate/deactivate/renew/delete API keys with masked display. Chat with AI about any teacher's feedback with smart history summarization for long conversations. Generate AI reports for individual teachers. Run the multi-step Setup Wizard to configure departments, semesters, sections, subjects, teachers, assignments, and students.

1.5 Goal and Objectives

The primary goal is to design, implement, and deploy an AI-enhanced student feedback platform bridging evaluation, analysis, and faculty coaching. 1. Implement role-based access control (RBAC) with JWT access/refresh token rotation ensuring secure, distinct interfaces for students, teachers, and administrators. 2. Develop an intuitive feedback submission engine with rating (1-10), comment input, version tracking, and progress indicators. 3. Integrate Google Gemini AI models (gemini-2.5-flash and gemini-2.5-flash-lite) for automatic teacher performance summaries from raw student comments. 4. Implement a multi-key rotation system with automatic invalid key detection and deactivation for high AI availability. 5. Design intelligent caching strategies for AI-generated summaries to minimize API latency and transaction costs. 6. Build an AI chatbot coaching interface for teachers and an admin-teacher chat with smart history summarization. 7. Provide administrators with granular student tracking by department with pagination and pending student identification. 8. Create a multi-step Setup Wizard with CSV bulk import for student records and automated conflict resolution. 9. Implement an AI Settings management panel for model selection and full API key lifecycle management. 10. Deploy the system using Vercel serverless architecture with MongoDB Atlas cloud database.

The scope encompasses the following implemented modules: Domain Configuration: Departments, semesters, sections, subjects, teacher assignments, student enrollments — all modeled as Mongoose schemas with proper relationships and indexes. Authentication Module: JWT-based secure session management with access token (short-lived) and refresh token (long-lived) rotation, bcrypt password hashing, and role-based authorization middleware. Student Portal: Assigned teacher cards, inline feedback form with rating slider (1-10) and comment textarea, progress tracking bar, feedback edit with version increment. Teacher Portal: Subject-wise analytics, rating distribution Recharts chart, per-subject breakdown table with color coding, AI-generated summary panel with stale indicator, AI Chat Coach with context-aware conversation. Admin Portal: Overview metrics (teachers, students, feedback count), teacher rankings bar chart, student tracking by department with pagination, pending students list across all departments, user CRUD management with password reset and cascading deletion, AI settings panel (model selection, key add/activate/deactivate/renew/delete with masked display), admin-teacher chat with history summarization, report generation. Setup Wizard: Multi-step wizard for creating departments, semesters, sections, subjects, teachers, assignments, students including CSV bulk import with conflict resolution for duplicate usernames and CNICs.

1.7 Limitations

External API Dependency: The AI services rely on Google Gemini API availability. Network latency or upstream interruptions can affect performance. API Quota Limits: Generation of summaries and chat responses is bound by Gemini rate limits. Mitigated by multi-key rotation and caching but extreme usage may result in throttling. Internet Connectivity: As a cloud-hosted platform utilizing MongoDB Atlas and external AI services, active internet is required. Language Limits: The AI engine parses qualitative data in English. Non-English comments may not be summarized with the same granularity.

AI Chat Coach with message history, typing indicator, and context-aware responses based on actual student comments. Admin Dashboard: Overview statistics (total teachers, total students, total feedback count). Teacher rankings table sorted by average rating with bar chart visualization. Student tracking by department with paginated table showing each student's submitted count, total assigned, completion status, and list of missing teacher names. Add/edit/delete users. Reset user passwords. Cascading deletion: deleting a student removes their feedback records and Student profile; deleting a teacher removes their teacher assignments and associated feedback. AI Settings Panel: Select between Gemini models: gemini-2.5-flash and gemini-2.5-flash-lite. Add API keys with label and masked display (4-char prefix + ... + 4-char suffix). Activate/deactivate keys. Renew keys (update API key value). Delete keys. View key status (active/inactive), last used timestamp, last failed timestamp, last error message. AI Multi-Key Rotation: Gemini service layer iterates through all active keys. On success: marks key with lastUsedAt. On failure: marks key with lastFailedAt and error message; if "API KEY NOT VALID" error, auto-deactivates the key. If all keys fail, throws appropriate error message guiding admin to configure keys. AI Summary Caching: Generated summaries are cached with feedback count. On subsequent load, if feedback count matches cache, cached summary is displayed instantly. If new feedback arrived, "stale" indicator prompts teacher to regenerate. Admin-Teacher Chat: Admin can chat with Gemini about any teacher's feedback. Chat builds context from actual student comments and ratings. Long chat history (>3000 chars) is automatically summarized via Gemini and compacted into a dense context. Supports Markdown formatting in responses. Setup Wizard: Multi-step configuration: Departments → Semesters → Sections → Subjects → Teacher Registration → Subject-Teacher Assignments → Student Registration → CSV Bulk Import. Each step has form validation and navigation controls. CSV Bulk Import: Parse CSV files with columns: name, username, password, email (optional), cnic (optional), phone (optional), sectionId, semesterId. Automatic conflict resolution for duplicate usernames and CNICs. Reports count of successful imports and conflicts. Theme Support: Dark/light theme toggle with persistent preference in context. Responsive design for mobile and desktop.

Operational Risks Addressed: Database connection failures in serverless environment — mitigated by Mongoose connection buffering. 3.5 Non-Functional Requirements

NFR1: API response times should be under 200ms for non-AI endpoints. NFR2: Cached AI summary retrieval should be under 50ms (from database). NFR3: AI summary generation should include a loading indicator with response typically under 5 seconds. NFR4: The system should handle multiple concurrent users through serverless auto-scaling on Vercel. NFR5: Database queries should be optimized with Mongoose indexes on frequently queried fields (userId, sectionId, teacherId, etc.). NFR6: Passwords must be hashed using bcrypt before storage. NFR7: JWT tokens must be signed with a secret key from environment variables. NFR8: AI API keys must be stored in the database and masked in the UI — never exposed in full.

CHAPTER 5: IMPLEMENTATION AND UNIT TESTING

5.1 Code Overview

5.1.1 Backend Architecture

The backend follows a modular MVC-inspired architecture with separate directories for models, routes, services, middleware, and utilities. The Express server is configured with CORS (allowing localhost:5173-5174, :3000, and *.vercel.app origins), JSON body parsing, and URL-encoded data parsing. Six route modules are mounted under /api/ with health check at /api/health. The server supports both traditional Node.js hosting and Vercel serverless deployment through conditional app.listen(). Mongoose connection is initialized asynchronously with query buffering enabled, allowing the server to start accepting requests even before the database connection is fully established. Database queries are queued and executed once the connection is ready.

5.1.2 Frontend Architecture

The frontend is built with React 18 using functional components and TypeScript. The project is structured into components (organized by role: admin, auth, common, setup, student, teacher), context providers (AuthContext for JWT management, ThemeContext for dark/light mode), API client (Axios instance with interceptors for token refresh), and top-level App.tsx with React Router v6 for client-side routing. The AuthContext manages user state, tokens, and provides login/logout/register functions. The Axios interceptor automatically checks token expiration and uses the refresh token endpoint (/api/auth/refresh-token) to obtain new access tokens transparently. ThemeContext provides dark/light mode toggling with persistent preference.

5.1.3 Database Schema Implementation (11 Models)

User: username (unique), password (bcrypt hashed), role (enum: student/teacher/admin), name, email. Used for authentication and RBAC middleware. Department: name, code. Root of the academic hierarchy. Semester: departmentId (FK), number (1-8), label (e.g., "Spring 2026"). Linked to department. Section: semesterId (FK), name (e.g., "A", "B"). Linked to semester. Subject: sectionId (FK), name, code. Linked to section. TeacherAssignment: teacherId (FK→User), subjectId, sectionId, semesterId. Links teacher to subject within section and semester. Student: userId (FK→User), sectionId, semesterId, rollNumber (unique), cnic (optional), phone (optional). Feedback: studentId (FK→Student), assignmentId (FK→TeacherAssignment), rating (Number 1-10), comment (String), version (Number default 1). Version increments on edit. AISummary: teacherId (FK), summaryText (String), feedbackCount (Number for cache invalidation). ChatSession: teacherId (FK), messages (array of {role, content, timestamp}). AISettings: selectedModel (enum), keys (array of {label, apiKey, isActive, lastUsedAt, lastFailedAt, lastError, createdAt, updatedAt}).

5.1.4 Gemini AI Integration Layer

The Gemini AI layer is implemented in backend/src/services/gemini.ts with the following key functions: runWithGemini: The core execution function that orchestrates multi-key rotation. It fetches all active keys from the AISettings collection, iterates through them attempting the API call, marks successful keys with lastUsedAt timestamp, marks failed keys with lastError and auto-deactivates keys that return "API KEY NOT VALID" errors. If all keys fail, it throws a descriptive error. generateTeacherSummary: Takes an array of student comments and ratings. Constructs a prompt with average rating, total responses, and individual comments. Sends to Gemini via runWithGemini. Returns the AI-generated text summary (3-4 sentences). chatWithTeacher: Takes message history, new message, and teacher feedback context as system instruction. Starts a chat session with history and returns AI response. System instruction enforces teacher-facing second-person tone and constructive coaching. chatWithAdmin: Similar to chatWithTeacher but optimized for admin-facing responses with Markdown formatting and concise answers. Context includes average rating and student comments. summarizeChatHistory: Takes long chat history and compresses it into a dense paragraph using Gemini. Used by admin-teacher chat when history exceeds 3000 characters to prevent token overflow.

5.1.5 Multi-Key Rotation & Auto-Disabling

The multi-key rotation system works as follows: 1. getRuntimeAIConfig() reads the AISettings document from MongoDB, extracting the selected model and all active keys. 2. runWithGemini() iterates through keys in order. For each key, it creates a GoogleGenerativeAI instance and attempts the operation. 3. On success: markKeySuccess() updates the key's lastUsedAt timestamp in the database. 4. On failure: markKeyFailure() updates the key's lastFailedAt and lastError. If the error indicates an invalid API key (detected via errorDetails reason "API_KEY_INVALID" or message containing "API KEY NOT VALID"), the key is automatically set to isActive: false. 5. If all keys fail and the last error was an invalid key error, the system throws: "All configured Gemini keys are invalid. Renew or add a valid key in Admin > AI Settings."

5.1.6 JWT Token Rotation Strategy

JWT authentication uses two token types: Access Token: Short-lived (configurable via env, default 15 minutes). Contains userId and role. Used for API authorization. Generated by generateAccessToken() in utils/jwt.ts using jwt.sign(). Refresh Token: Long-lived (configurable via env, default 7 days). Contains userId. Used only at the /api/auth/refresh-token endpoint. Generated by generateRefreshToken(). On login, both tokens are returned. The frontend stores them and the Axios interceptor checks access token expiry before each request. If expired, it calls the refresh endpoint with the refresh token, gets a new access token, and retries the original request.

5.1.7 AI Summary Caching Strategy

The summary caching system (in services/summary.ts) works as follows: 1. When a teacher requests their AI summary, the system checks the AISummary collection for an existing record. 2. If a cached summary exists and the stored feedbackCount matches the current count of feedback records, the cached summary is returned immediately (no AI call). 3. If the feedbackCount doesn't match (new feedback was submitted), the summary is marked as "stale" — the teacher sees a badge indicating new feedback is available. 4. When the teacher clicks "Generate Summary", the system fetches all fresh feedback, calls Gemini, saves the new summary with the updated feedbackCount, and returns the result. This strategy reduces AI API costs by approximately 90% for frequent summary views.

5.1.8 Chat History Summarization

The admin-teacher chat feature includes smart history management: 1. The system tracks the total character length of the conversation history. 2. If history exceeds 3000 characters, the system triggers summarizeChatHistory() which sends the entire conversation to Gemini with a prompt asking for a dense single-paragraph summary. 3. The original history is replaced with a compacted history containing the summary as a system message: "[SYSTEM: PREVIOUS CHAT COMPACTED] We previously established: {summary}". 4. The compacted history is sent back to the frontend so it can replace its local state. This prevents token overflow in long conversations while preserving context.

5.1.9 CSV Bulk Import Engine

The CSV import feature (services/csv.ts + setup routes) works as follows: 1. Administrator uploads a CSV file via Multer middleware (file buffer). 2. The parseCsvBuffer() function uses csv-parse/sync to parse the buffer with columns: true (first row as headers), skip_empty_lines: true, trim: true. 3. The parsed rows include fields: name, username, password, email (optional), cnic (optional), phone (optional), sectionId, semesterId. 4. For each row, the system checks for duplicate username and duplicate CNIC against existing User and Student records. 5. Non-duplicate records are created as User documents (hashed password) and Student documents (with generated rollNumber). 6. The system returns a report: total rows, successfully imported count, and conflicts with details.

5.1.10 User Cascading Deletion

When an administrator deletes a user, the system performs cascading cleanup: Deleting a Student: The system finds the Student profile by userId, deletes all Feedback records linked to that student's _id, deletes the Student profile, then deletes the User document. Deleting a Teacher: The system finds all TeacherAssignment records for that teacher, collects their assignmentIds, deletes all Feedback records linked to those assignments, deletes the TeacherAssignment records, then deletes the User document. This ensures no orphaned records remain in the database after user deletion.

5.1.11 Teacher Ranking Algorithm & Admin Report Logic

Teacher Rankings (GET /api/admin/teachers/rankings): The ranking algorithm computes teacher performance by aggregating all feedback data per teacher. The backend queries all users with role "teacher", finds their TeacherAssignment records, collects all associated Feedback records, averages the rating values to produce averageRating, and sorts descending by averageRating. The frontend displays these with color-coded ratings (green ≥7, yellow 5-6.9, red <5), medal emojis for top 3, and an expandable AI Intelligence Report with chat functionality. Admin Report Generation (POST /admin/report/generate/:teacherId): This endpoint triggers AI summary generation by calling generateAndCacheSummary() from the summary service. It fetches the teacher's assignments, gathers associated feedback comments and ratings, constructs a Gemini prompt including the average rating and response count, sends it via runWithGemini with multi-key rotation, and caches the result with the current feedback count for instant future retrieval. Admin Chat with AI (POST /admin/report/chat/:teacherId): Enables administrators to have contextual AI conversations about any teacher's feedback. It fetches assignments and feedback, calculates the average rating, and constructs a system prompt instructing the AI to act as an "expert University HR Performance Evaluator." The context includes formatted student comments and ratings. If chat history exceeds 3000 characters, summarizeChatHistory() compacts it into a dense paragraph. Responses use Markdown formatting (bold, bullet points) rendered by the FormattedMessage component.

5.1.12 Frontend API Client Configuration

The frontend API client (src/api/client.ts) is built on Axios with the following configuration: 1. Base URL: Configured via VITE_API_URL environment variable, defaulting to the production API at https://studentfeedbackicp-api.vercel.app/api. 2. Request Interceptor: Automatically attaches the JWT access token from localStorage as a Bearer token in the Authorization header for every request. 3. Response Interceptor (Token Refresh): On 401 responses, the interceptor checks if the request was already retried. If not, it sends the refresh token to /auth/refresh-token, stores the new access token, and retries the original request. If the refresh fails (token expired/invalid), it clears localStorage and redirects to /login. FormattedMessage Component (src/components/common/FormattedMessage.tsx): This component renders AI responses containing Markdown-like formatting. It splits text by newlines, detects bullet points (lines starting with "- " or "* ") and wraps them in

    lists, detects and renders bold text (**text**) as tags, and wraps remaining lines in

    tags with whitespace preservation. This gives the AI responses a clean, formatted appearance without requiring a full Markdown parser.
